# Принципы работы фронта

## 1. Архитектурная независимость

Фронтенд является **отдельным проектом**:

- имеет собственный репозиторий;
- развивается независимо от backend;
- может деплоиться отдельно;
- не содержит бизнес-логики сервера.

Фронт не привязан к конкретной реализации backend.

---

## 2. Работа с любым API

Фронт может быть настроен для работы с **любым API**, если оно:

- соответствует установленному формату меню;
- возвращает стандартизированные конфиги;
- придерживается согласованной структуры ответов (`list`, `show`, `create`, `update` и т.д.);
- использует одинаковые принципы описания `options`, `fields`, `columns`, `actions`, `filters`.

Фронт — это универсальный UI-движок, который:

- получает описание интерфейса через API,
- динамически строит страницы,
- может работать как в полностью динамическом, так и в статическом режиме.

---

## 3. Первоначальный конфиг

Для различных сущностей используются конфигурационные файлы в формате **PHP-массивов Laravel**.

Эти конфиги описывают:

- пункты меню;
- разделы;
- поля;
- настройки отображения;
- действия;
- фильтры;
- сортировку и ограничения.

Структура соответствует документу:

> «Динамическое формирование фронта»

### Важный принцип

Фронт **может иметь жёстко определённые сущности, их параметры и конфиги**.

Это означает:

- часть сущностей может быть заранее известна фронту;
- их структура может быть зафиксирована на стороне фронта;
- API в этом случае поставляет только данные.

---

## 4. Источник конфигурации и приоритеты

Фронт может получать описание разделов двумя способами:

1. Из локальных конфигурационных файлов (PHP-конфиги).
2. Из API (например, `/api/menus/list`).

### Приоритет источников

**Конфиги являются первичными.**

Если для раздела существует локальный конфиг:

- фронт использует именно его;
- запрос к `/api/menus/list` для получения настроек этого раздела не выполняется;
- настройки из API игнорируются.

Если локального конфига нет:

- фронт получает описание раздела через `/api/menus/list`;
- раздел строится динамически на основании данных API.

---

### Возможные режимы работы

1. **Полностью статический режим**
   - все разделы описаны локально;
   - API используется только для данных.

2. **Полностью динамический режим**
   - конфигов на фронте нет;
   - вся структура приходит через `/api/menus/list`.

3. **Гибридный режим**
   - часть разделов описана локально;
   - часть подтягивается из API.

---

## 5. Загрузка иконок и медиа по сущностям

Для получения иконок, флагов, галочек и других визуальных ресурсов используется контроллер **Download**.

### URL-формат

Ресурс запрашивается по пути:

```
/download/image/{раздел}/{поле}/{настройка}/{значение}
```

Где:

- `раздел` — имя раздела/сущности (например `clicks`);
- `поле` — поле, к которому относится ресурс (например `country`);
- `настройка` — тип ресурса или вариант отображения (например `flag`, `icon`);
- `значение` — конкретное значение поля (например `MY`).

### Пример

Флаг Малайзии:

```
/download/image/clicks/country/flag/MY
```

---

## 6. Связь с модификаторами отображения

Если для поля установлен модификатор:

- `modifier: flag`
- `modifier: icon`

то фронт формирует `src` для `<img>` по стандартному пути:

```
/download/image/{раздел}/{поле}/{modifier}/{значение}
```

Таким образом:

- фронт не хранит локальные маппинги иконок;
- все изображения запрашиваются через единый endpoint;
- визуальное представление стандартизировано.

---

## 7. Поведение при отсутствии изображения

Если запрошенное изображение не найдено:

- контроллер возвращает **дефолтное изображение**, соответствующее полю и типу ресурса;
- фронту не требуется реализовывать fallback-логику;
- подмена ресурса полностью инкапсулирована на стороне backend.

Это обеспечивает:

- устойчивость интерфейса;
- отсутствие ошибок отображения;
- централизованное управление ресурсами.
